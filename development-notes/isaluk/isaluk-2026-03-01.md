# Дата: 2026-03-01

- **Что было сделано:** Ознакосилась с базовой настройкой проекта, выполненной тимлидом. Впервые столкнулась со сборщиком Vite, поэтому предварительно изучила материалы ("Быстрый старт с Vite"). Взяла в работу архитектурную задачу Issue #22: architecture: implement core Store - phase 1. Реализовала функциональную фабрику `createStore` на основе паттерна Observer на Vanilla TypeScript и открыла Pull Request (Closes #22).
- **Проблемы:** 1. _Архитектурный контракт:_ Требовалось обеспечить строгую приватность состояния (Private State), чтобы оно не было глобальной переменной. Использование привычного синтаксиса классов (class Store) создавало риск прямого доступа к свойствам. 2. _Строгая типизация:_ При реализации метода обновления состояния нужно было заставить TypeScript понимать, что мы передаем не весь объект целиком, а только изменяемую его часть, при этом не нарушая интерфейс `AppState`.
- **Решения:**
  1. _Инкапсуляция через замыкания:_ Я отказалась от классов в пользу функционального подхода. Состояние живет исключительно внутри области видимости (scope) функции `createStore`. Доступ к нему возможен только через `getState()`, а изменение — только через иммутабельный `setState()` с использованием spread-оператора.
  2. _Решение проблемы с типами:_ Для метода `setState` я применила утилиту `Partial<AppState>`. Это решило проблему строгой типизации при частичном обновлении свойств.
  3. _Оптимизация Observer:_ Для хранения функций-подписчиков (listeners) использована структура `Set`. Это гарантирует отсутствие дубликатов и обеспечивает удаление подписки за время O(1). Метод `subscribe` возвращает функцию очистки (unsubscribe), что защитит наше SPA от утечек памяти при размонтировании компонентов.
