## 1. Описание работы над проектом

В рамках проекта «DevQuest» я сфокусировалась на проектировании надежной, событийно-ориентированной архитектуры и реализации функционала приложения на **Vanilla TypeScript** со сборщиком Vite. Моей главной целью было избежать хаоса, свойственного ванильным SPA, и создать предсказуемую систему управления состоянием и маршрутизацией.

**Что было сделано:**

- **Архитектура с нуля:** Я отказалась от глобальных переменных в пользу инкапсулированного State Manager'а, построенного на паттерне Observer. Также были внесены изменения в Router связанные с защитой маршрутов (Route Guards) по принципу "Белого списка" (Zero Trust) и глобальная шина событий (EventBus) для общения между компонентами.
- **Синхронизация и хранение данных:** Спроектировала базу данных в Supabase (BaaS) с настройкой Row Level Security (RLS). Разработала Offline-First подход: гибридную систему сохранения прогресса (localStorage + облако), которая защищает данные игрока при обрыве сети.
- **UI/UX:** Реализовала сложные экраны (Onboarding/Auth) с кастомной валидацией (debounce), выбором аватаров и глобальным компонентом Loader для блокировки интерфейса во время асинхронных операций, а также самостоятельный элемет TaskTimer.
- **Интеграция AI & Backend:** Создала и задеплоила Edge Function в Supabase на Deno. Она выступает в роли безопасного прокси для запросов к Google AI SDK (Gemini), скрывая API-ключи от клиента и возвращая динамические игровые задачи.

**Главные сложности и их решения:**

- _Проблема "холодного старта" и обновления по F5:_ Роутер срабатывал быстрее, чем подтягивалась сессия из базы, выбрасывая авторизованного игрока на стартовый экран. Решение: внедрение паттерна `bootstrap` с синхронным ожиданием `getSession()` перед рендером и флагом готовности приложения (`isReady`).
- _Рассинхронизация вкладок браузера:_ Игрок мог проходить игру параллельно в двух вкладках, ломая сохранение. Решение: использование `BroadcastChannel` для реактивной синхронизации состояния между вкладками и мгновенного форс-редиректа "отставших" экранов.
- _Утечки памяти и сброс таймеров:_ Стандартный `setInterval` троттлился в неактивных вкладках. Решение: привязка таймера к `Date.now()` и строгая очистка листенеров при смене маршрута.

## 2. Личные Feature Components (Разработано без AI)

Это те компоненты, в которых я реализовала самую сложную логику проекта и готова детально разобрать их на защите:

1. **Custom State Manager & Routing System (Архитектура состояния)**
   - **Сложность:** Необходимо было обеспечить строгую приватность состояния и реактивное обновление UI без лишних перерисовок, приводящих к потере фокуса в инпутах.
   - **Решение:** Функциональная фабрика `createStore` на основе паттерна Observer (использование замыканий и структуры `Set` для O(1) отписок). Внедрена умная прослойка в Router, которая сравнивает сериализованные маршруты и блокирует перерисовку DOM при фоновых обновлениях токена. Решена проблема гонки данных (Race Condition) при инициализации приложения.
   - **Ссылки:** [PR #36](https://github.com/OlgaMinaievaWebDev/rs-tandem-devquest/pull/36), [PR #60](https://github.com/OlgaMinaievaWebDev/rs-tandem-devquest/pull/60), [PR #75](https://github.com/OlgaMinaievaWebDev/rs-tandem-devquest/pull/75)

2. **Core Game Loop & EventBus (Ядро игрового цикла)**
   - **Сложность:** Избежать жесткой связности (Prop Drilling) между глубоко вложенными компонентами UI и бизнес-логикой игры (таймеры, счетчики дней, проверка условий победы).
   - **Решение:** Разработка строго типизированного `EventBus` (Pub/Sub) с использованием TypeScript generics. Создание контроллера `DayManager`, который подписывается на события интерфейса (`TASK_FINISHED`) и независимо управляет правилом "2 из 3 задач". Внедрен Widget Engine для отрисовки мини-игр (напр. Quiz) и независимый TaskTimer, устойчивый к фоновому режиму браузера.

3. **Task Timer & Интерактивный Quiz Widget**
   - **В чём суть:** Ядро геймплея — виджет для прохождения тестов с визуальным откликом, привязанный к таймеру задач.
   - **Техническая реализация:**
   - **Защита от троттлинга браузера:** `TaskTimer` написан с опорой на `Date.now()`, а не просто на счетчик внутри `setInterval`. Это гарантирует абсолютную точность таймера, даже если пользователь свернул вкладку браузера (браузеры искусственно замедляют таймеры в неактивных вкладках).
   - **Слабая связность (Decoupling):** Оба компонента не мутируют глобальное состояние напрямую. Они общаются с игровыми менеджерами исключительно через отправку сообщений в `EventBus` (`TASK_FINISHED`), что делает их переиспользуемыми и тестируемыми.
